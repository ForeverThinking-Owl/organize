# 工具、人与组织（二）——实践主体与实践主权的分离

---

## 摘要

AI 时代的关键变化，在于**实践主体与实践主权开始分离**。

AI 可以参与何时、何地、何事、何因、何人、如何实践的判断，成为操作意义上的实践主体；人和组织则保留目标、边界、例外、审批、监督与责任。

- **实践主体**负责运行
- **实践主权**负责治理

由此，审批成为实践主权的最小机制，监督成为实践主权的持续形式，责任成为实践主权的最终形式。

---

## 一、实践曾经由人完整承担

在最初的实践中，人同时承担两件事：**运行与负责**。

人感知对象，判断处境，选择工具，完成行动，观察结果，并承受由此带来的后果：

- 谁拿起石器，谁完成切割
- 谁点燃火种，谁承担取暖、烹饪、防御的结果
- 谁治疗病人，谁面对诊断、用药和预后的责任
- 谁答复客户，谁承担承诺、解释和后续处理的压力

在这种状态下，**实践主体与实践主权高度重合**。

| | 实践主体 | 实践主权 |
|------|----------|----------|
| **回答的问题** | 谁在行动？谁在判断？谁在执行？谁在反馈？ | 为什么行动？边界在哪里？风险由谁确认？后果由谁承担？规则由谁修正？ |

当一个人独自完成某项实践时，这两组问题都落在同一个人身上。人既是实践运行者，也是实践责任者。人既决定怎么做，也承担做完之后的结果。

> 这种重合，构成了传统实践的基本形态。

---

## 二、工具使实践能力外化

工具的出现，使实践能力开始离开身体。

- **刀**延伸切割能力
- **火**延伸能量转化能力
- **文字**延伸记忆能力
- **账簿**延伸核算能力
- **机器**延伸力量能力
- **计算机**延伸信息处理能力

工具把原本依赖身体、经验和即时判断的实践方式外化出来，使它们可以被保存、重复、传递和组合。

> **工具带来了第一个分离：实践能力与人的身体开始分离。**

人仍然选择工具、使用工具、承担工具使用的后果，但工具已经替人稳定了一部分实践方式。石器把切割稳定下来，文字把记忆稳定下来，机器把力量稳定下来，软件把信息处理稳定下来。人的能力通过工具扩展，实践也因此获得更大的尺度和复杂度。

工具使人能够完成身体无法直接完成的转化。

这种外化带来了一个重要结果：实践开始由"人直接做"转向"人借助工具做"。人的实践能力因此变成一种组合能力——人开始围绕对象选择工具、调整顺序、判断风险、评估结果。

> 工具扩展了实践主体的能力，但实践主权仍然主要由人承担。人仍然决定目标，判断边界，承受后果。

---

## 三、组织使实践主体分布化

当实践越来越复杂，单个人的技能边界开始显现。一个人可以掌握工具，但难以掌握所有工具；一个人可以完成局部实践，但难以持续完成复杂社会生产。

于是，不同人的不同技能被协同起来：

- 医疗实践中，有诊断、护理、检验、手术、药剂和管理
- 制造实践中，有设计、采购、加工、质检、仓储和交付
- 客户服务实践中，有沟通、查询、售后、财务、技术和反馈

每一种复杂实践，都要求不同技能围绕同一个对象形成协同。**组织由此出现为一种稳定的协同结构。**

在组织中，实践主体已经开始分布化。一次完整实践往往由多个人、多种工具、多项技能共同完成。客户投诉可能由客服角色接收，由售后角色判断政策，由财务角色审查退款，由技术角色排查故障，由开发角色修复缺陷，最后再由客服角色统一回应。

> 此时，实践主体已经从单个人变成协同结构。

实践主权也开始结构化。组织通过角色、流程、权限、审批、规章和责任，把实践主权分配到不同层级。某些人负责执行，某些人负责复核，某些人负责审批，某些人负责规则制定，某些人承担最终责任。

> **这就是组织带来的第二次分离：实践运行开始分布在多个主体之间，实践主权通过规则与责任结构被保留下来。**

---

## 四、AI 使实践主体获得可运行形态

AI 的出现，使这种分离进入新的阶段。

AI 已经具备一定的实践判断能力。它能够在语言、情境、规则和历史信息中判断：何时需要行动，行动发生在哪里，当前对象属于何种实践，为什么应当如此处理，应该由谁承担，以及如何展开步骤。

**这就是实践的 5W1H：**

| 维度 | 问题 |
|------|------|
| **When** | 何时实践 |
| **Where** | 何地实践 |
| **What** | 何事实践 |
| **Why** | 何因实践 |
| **Who** | 何人实践 |
| **How** | 如何实践 |

过去，这些判断主要由人完成。人识别问题，人判断场景，人决定调用哪个角色，人选择工具，人推动下一步。AI 出现之后，这些常规判断开始可以被语言和情境调度。

**例如，面对一句话：**

> 客户说扫码枪连不上系统，还要求退款。

AI 可以判断其中包含多个实践对象：

```
设备连接问题
售后问题
退款风险
客户投诉
潜在技术缺陷
```

它可以判断实践路径：

```
客服角色先分流
  → 技术角色排查设备连接
  → 售后角色判断退换修政策
  → 财务角色审查退款请求
  → 必要时开发角色分析缺陷
```

它也可以判断治理点：

```
涉及退款 → 需要审批
涉及外部回复 → 需要确认
涉及技术缺陷 → 需要创建开发任务
```

> AI 因此开始成为操作意义上的实践主体。它能够感知语义对象，解释情境，选择工具，编排步骤，触发协同，接收反馈，并继续下一步。

工程上，这种实践主体可以被称为 **Actor**。

Actor 具备：

| 属性 | 含义 |
|------|------|
| Identity | 身份 |
| Responsibility | 职责 |
| Memory | 记忆 |
| Permission | 权限 |
| Approval Judgment | 审批判断 |
| Skill | 技能 |
| Tool Scope | 工具范围 |
| Runtime State | 运行状态 |
| Trace | 运行痕迹 |

> Actor 的意义，在于把 AI 从抽象模型转化为有边界、有职责、有工具、有记忆、有治理关系的实践运行单元。

---

## 五、实践主体与实践主权开始分离

AI 成为实践主体，并不意味着 AI 拥有实践主权。

这里需要做一个清晰区分：

| | 实践主体（运行） | 实践主权（治理） |
|------|-------------|-------------|
| **处理** | 何时启动、进入哪个场景、识别什么对象、调用什么技能、使用什么工具、如何执行、如何反馈、如何进入下一步 | 目标是否正确、边界是否清晰、风险是否可接受、例外如何裁决、结果由谁负责、规则如何修正 |

AI 可以承担越来越多的实践调度。它可以判断何时实践、何地实践、何事实践、何因实践、何人实践、如何实践。它可以调用工具，生成行动路径，协调不同角色，输出结构化结果。

人和组织必须保留实践主权。目标、边界、例外、责任和价值判断，仍然需要由人和组织承担。

> **AI 获得实践调度权，人和组织保留实践主权。**
>
> **实践主体与实践主权开始分离。**

这不是人的消失，而是人的位置上移：

| 过去 | 未来 |
|------|------|
| 常规执行者 | 目标设定者 |
| 查询者 | 边界制定者 |
| 录入者 | 风险审批者 |
| 催办者 | 例外裁决者 |
| 复制者 | 规则修正者 |
| — | 最终责任者 |

> AI 处理实践如何运行。
> 人决定实践为何值得、何处停止、谁来负责。

---

## 六、审批是实践主权的最小机制

审批常常被理解为流程节点，但在 AI 实践系统中，审批具有更深的含义。

> **审批是实践主权的最小机制。**

当 AI Actor 准备执行某个高风险动作时，系统必须暂停，等待主权确认。例如：

- 是否允许创建退款申请
- 是否允许发送外部回复
- 是否允许修改订单
- 是否允许提交报价
- 是否允许部署代码
- 是否允许写入财务系统

这些动作一旦执行，就会进入社会生产并产生后果。**它们需要主权确认。**

审批不是简单的"同意"或"拒绝"。审批至少包含五种能力：

1. **确认目的**
2. **限制边界**
3. **修改参数**
4. **要求补充信息**
5. **承担责任**

因此，一个完整审批并不只是点击按钮。它可能表现为：

```
同意执行
拒绝执行
修改金额后同意
脱敏结果后释放
添加责任意见
升级给更高层级
要求补充证据
```

在工程上，审批发生在 ToolCall 生命周期中：

| 阶段 | 场景 | 重点 |
|------|------|------|
| `before_call` | 执行前确认 | 写入类 Tool |
| `after_call` | 结果释放前确认 | 查询类 Tool |
| `before_writeback` | 最终写回前确认 | 外部回复、正式写回 |

> 审批让 AI 的实践主体性进入人的主权边界。

---

## 七、监督是实践主权的持续形式

审批是点状的，监督是连续的。

实践主权不能只在高风险节点出现。AI 参与实践之后，运行过程必须可观察、可追踪、可复盘、可修正。否则，人虽然名义上保留主权，却无法理解系统如何行动，也无法有效承担责任。

**监督需要 Trace。**

Trace 记录：

```
输入是什么
Actor 看到了什么上下文
Actor 调用了什么 Skill
Skill 选择了什么 Tool
ToolCall 参数是什么
权限如何判断
审批如何发生
工具返回了什么
最终输出是什么
哪些记忆被生成
哪些规则需要修正
```

**监督还需要记忆。**

一次实践的结果不应只停留在日志中。它应该被整理为经验，成为下一次实践的条件。例如：

- 某类退款请求经常需要财务复核
- 某类技术故障经常导致客户投诉
- 某个 Actor 经常生成越界回复
- 某个审批人经常修改某类参数
- 某个工具返回结果需要脱敏后释放

这些经验进入记忆之后，实践系统才能逐渐改进。

> **监督不是事后检查。监督是实践主权的持续形式。**它使人和组织能够持续观察 AI 实践主体，并通过反馈、记忆和规则修正，让系统变得更可靠。

---

## 八、责任是实践主权的最终形式

实践最终要面对后果。

- 一次错误退款 → 影响资金
- 一次错误回复 → 影响客户关系
- 一次错误诊断 → 影响治疗
- 一次错误部署 → 影响系统稳定
- 一次错误审批 → 影响组织信用

AI 可以执行，AI 可以判断，AI 可以调度，但社会实践的后果必须有责任结构承接。

责任回答的是：

```
谁设定了目标？
谁授权了边界？
谁批准了风险？
谁接受了结果？
谁修正了规则？
谁面对外部后果？
```

> 没有责任结构的自动运行，只是自动化。
> 有责任结构的自动运行，才可能成为可治理的实践。

**责任是实践主权的最终形式。**

审批确认风险，监督观察过程，责任承接后果。三者合起来，构成 AI 实践系统中的主权结构。

在工程上，每一次关键行动都应保留：

```
responsible_actor
approver
approval_decision
tool_call_log
trace
memory_candidate
policy_version
```

> 实践主体可以是 AI Actor。
> 实践主权必须落实到人和组织的责任结构中。

---

## 九、Actor 是实践主体的工程形态

将上述概念落到工程系统中，Actor 可以被定义为实践主体的工程形态。

Actor 不是一个普通聊天模型。Actor 是具有身份、职责、记忆、权限、审批判断、Skill、Tool 范围和运行状态的实践运行单元。

**Actor 具备：**

| 属性 | 问题 |
|------|------|
| **Identity** | 我是谁 |
| **Responsibility** | 我负责什么 |
| **Memory** | 我记得什么 |
| **Permission** | 我能看什么、做什么 |
| **Approval Judgment** | 我何时需要请求审批，何时可以审批别人 |
| **Skill** | 我如何完成某类实践 |
| **Tool** | 我能调用哪些实践入口 |
| **Trace** | 我留下什么运行痕迹 |

**Actor 的每一步输出都应是结构化决策：**

```
tool_call
request_approval
handoff
ask_human
final_output
memory_candidate
```

系统在 ActorDecision 之后执行权限校验、审批判断、工具执行和结果过滤。这样，AI 的判断能力可以进入实践，同时被主权结构约束。

| 概念 | 工程形态 |
|------|----------|
| **实践主体** | Actor |
| **实践主权闸门** | Approval |
| **主权监督痕迹** | Trace |
| **主权修正材料** | Memory |

---

## 十、结语：谁来实践，谁来负责

AI 时代的核心变化，在于**实践主体与实践主权开始分离**。

- **实践主体**回答"谁来运行"
- **实践主权**回答"谁来决定、谁来停止、谁来负责"

AI 可以逐渐承担操作意义上的实践主体角色。它能够识别对象、判断情境、选择工具、编排步骤、调度协同、接收反馈，并推动常规实践自动运行。

人和组织保留规范意义上的实践主权。目标、边界、例外、审批、监督和责任，仍然必须由人和组织承担。

由此，未来组织运行的核心问题发生变化：

```
过去：人如何使用工具完成实践
现在：AI 如何作为实践主体参与运行
关键：人如何保留实践主权
```

> 实践主体处理如何运行。
> 实践主权处理为何值得、何处停止、谁来负责。

> **AI 接管实践调度，人保留实践主权。**
> **这是工具、人与组织在 AI 时代的新分工。**
